# Asset brief — Split-rail park-fence kit

Implements backlog idea #2 in [../park_asset_ideas.md](../park_asset_ideas.md): a
split-rail fence kit that shapes paths and gives partial, readable cover without
glowing blockout strips. Setting reference: [../../setting.md](../../setting.md),
which names split-rail fences as canonical Bouneurmaum National Park infrastructure.

## Goal

A modular, weathered post-and-rail (split-rail) fence kit built to real meters with
exact mesh collision, sharing one set of materials. Straight runs tile cleanly; a
heavy corner/end post turns or caps a run; a working timber gate fills a gap.

## Pieces (3 prefabs, 4 models)

| Prefab | Model(s) | Role |
|--------|----------|------|
| `FenceSplitRail` | `fence_split_rail.vmdl` | Tileable straight bay: one mortised post + three split rails over a 3 m bay. The core repeatable module. |
| `FenceSplitRailCorner` | `fence_split_rail_corner.vmdl` | Heavy standalone post with rail stubs on two perpendicular faces — turns a run 90° or caps/terminates it (also the end post). |
| `FenceSplitRailGate` | `fence_split_rail_gate_frame.vmdl` (static) + `fence_split_rail_gate_leaf.vmdl` (swinging) | Two sturdy gate posts flanking an opening, plus a braced field-gate leaf that swings open on player proximity. |

## Style

Sun-greyed hand-split timber: square hewn posts with shallow pointed (water-shed)
tops, rounded-octagon split rails with a slight sag and riven facets, flat-shaded
rough-hewn look. The gate leaf is a braced field gate (hinge + latch stiles, three
rails, a diagonal brace) with dark metal hinge straps and a latch.

## Materials (`Assets/materials/environment/`)

- `fence_wood.vmat` — weathered grey-brown split timber (posts, rails, gate frame
  and leaf). metalness 0, rough 0.85. Material slot `Wood`.
- `fence_endgrain.vmat` — sawn/split cut ends (rail ends, post tops, stub ends).
  metalness 0, rough 0.82. Material slot `Endgrain`.
- Gate hinge/latch hardware reuses `house_small_metal.vmat` — material slot `Metal`,
  only the gate leaf uses it.

Textures are generated by `scripts/generate_park_fence_textures.py` (numpy+PIL,
tileable, IDAT chunked ≤64 KB so s&box decodes them — see the log_cabin grey-texture
bug). Grain runs along V (length); tiling is baked into the UVs so the vmats stay at
`g_vTexCoordScale [1,1]`. Blender material slot names must match the remap keys
exactly: `Wood`, `Endgrain`, `Metal`.

## Pipeline / scale / collision

- Builder: `scripts/build_park_fence.py` — `build_park_fence("straight"|"corner"|
  "gate_frame"|"gate_leaf")`.
- Configs: `scripts/fence_split_rail*_asset_pipeline.json` (`global_scale 0.0254`,
  axis `-Y` fwd / `Z` up, `render_mesh` collision, `surface_prop wood`).
- Real meters: posts stand along Blender **+Z**, rails run along **+X**, bottom at
  **Z=0** — EXCEPT the gate leaf, whose origin is the **hinge** at ground (0,0,0)
  so the vmdl pivots about the hinge like `log_cabin_door`. Prefab `Visual.Scale =
  15.5` → real size (39.37 units/m).
- Exact mesh collision: `PhysicsMeshFile` nested in `PhysicsShapeList`; each prefab
  has ONE `ModelCollider` on its `Visual` child (same GameObject as the
  `ModelRenderer`). Gaps between rails stay open. See
  [../../collision_authoring.md](../../collision_authoring.md).

## Gate mechanism

`FenceSplitRailGate` adds a `Gate` child (mirrors the log-cabin `Door`): a
`ModelRenderer` + `ModelCollider` with **`Static = false`** (so the closed leaf
blocks and stays solid at any open angle) and a `DroneVsPlayers.SwingDoor`
(`AutoOpenOnProximity`). The leaf has no `Rigidbody` — SwingDoor keyframes it. If
the leaf swings the wrong way in a playtest, negate `SwingDoor.OpenAngle`.

## Placement notes

Posts/frame rest on Z=0, so they sit on flat terrain by default; lower/rotate per
instance on slopes. **Tiling:** the straight bay has one post (at its origin) with
rails cantilevering +X across the 3 m bay, so placing bays end-to-end at 3 m gives a
single post per joint and continuous rails (no z-fighting doubles). Cap or turn a
run with `FenceSplitRailCorner`. Prefabs appear under **Drone vs Players /
Environment / Fences** in the create menu. Place via the prefab (never the raw
`.vmdl`, which would spawn a dynamic Prop that falls).

### Laying a long run quickly — `FenceLine`

`Code/Game/FenceLine.cs` (a `Component.ExecuteInEditor`) generates an entire run in
one step. Put it on an empty GameObject, assign the `FenceSplitRail` prefab to
`SectionPrefab`, and set `Count`. It clones the section along the object's local +X at
`Spacing` (default **118** units ≈ one 3 m bay), so move/rotate the one object to
position and aim the whole fence; `CurvePerSection` (deg) sweeps it into an arc, and
`EndPostPrefab` (e.g. `FenceSplitRailCorner`) caps the far end. Generated sections are
`NotSaved` and regenerated each load (the component is the source of truth — no scene
bloat); press **Bake to scene** to freeze them into permanent, individually editable
instances and disable the builder.

Easiest entry point: drag **`FenceLineBuilder.prefab`**
(`Assets/prefabs/environment/FenceLineBuilder.prefab`, menu *Environment / Fences / Fence
Line (auto run)*) straight from the asset browser — it's an empty object with `FenceLine`
already wired (`SectionPrefab` → FenceSplitRail; no end post by default), so a full run of
sections appears the moment it's dropped. Assign `EndPostPrefab` (e.g. FenceSplitRailCorner)
yourself if you want a far-end cap. A prefab `GameObject` property is serialized as
`{ "_type": "gameobject", "prefab": "<path>" }` (same as DroneDeployer's drone prefabs).
Note the section is faced `heading - 90°`: the split-rail bay's rails run along the model's
local **+Y** (Blender +X → s&box +Y on export), so they must be turned to lie along the run.
The no-code alternative is still available: place one `FenceSplitRail` and Alt/Ctrl-drag-
duplicate it with grid snapping set to ~118 units.
